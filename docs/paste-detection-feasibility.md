# java-quest コピペ検知 実現性検討メモ v0.1（Draft）

策定日: 2026-04-14
策定者: Vera（AIアシスタント）

本ドキュメントは issue #1 フィードバック #1-2「コピペ検知機能」の実現性を検討した **調査メモ** である。実装方針を確定するものではなく、CEO 判断の材料として採用候補・却下案とその理由を整理する。

---

## 0. 検討の背景

### 0.1 要望

> 学習効果のため、コピペ検知があるとよい。実現性は要検討。（issue #1）

### 0.2 設計方針

`tamper-detection.md` と同じく **善意推定 / 学習体験の保護** を前提とする。冒険者を疑うより、学習行動を自然に誘導する設計を優先する。

### 0.3 前提

- java-quest は Claude Code の Skill として動作する
- 冒険者は自身の IDE / エディタでコードを書き、ファイルとして提出する
- Skill 側から IDE のキーストロークや編集履歴にはアクセスできない
- AI補助（Copilot 等）の OFF 推奨は #1-5 で別途対応済み

---

## 1. 技術的に検知できること / できないこと

| 観点 | 可否 | 根拠 |
|------|------|------|
| キーストローク・タイピング速度 | ❌ | Skill から IDE 入力イベントは見えない |
| クリップボード監視 | ❌ | 権限・プライバシーの両面で不適切 |
| IDE の編集履歴取得 | △ | VS Code 拡張等を別途作れば可能だが、Skill 単独では不可。スコープ外 |
| 提出コードのスナップショット | ✅ | 既存のフロー（ex-NNN/ , boss/ ）で取得可能 |
| 提出までの経過時間 | ✅ | `sessions.log` の `dungeon_enter` / `exercise_result` 差分で算出可能 |
| コード本文の AI 解析 | ✅ | 既存の採点 AI を流用できる |

**結論**: 「直接的なコピペ行為の検知」は原理的に不可能。代替として **提出物と行動の観察による推定** のみ実現可能。

## 2. 評価したアプローチ

| # | アプローチ | 概要 | 評価 |
|---|-----------|------|------|
| A | 所要時間ヒューリスティック | `dungeon_enter` → `exercise_result` の経過時間が閾値未満なら要注意 | △ |
| B | コード様式の一貫性チェック | 同一冒険者の過去提出と比較し、突然 idiomatic / 高度になったら要注意 | △ |
| C | カリキュラム先取り検知 | 学習前の構文（Stream API, try-with-resources 等）が出現したら要注意 | ○ |
| D | AI 生成物の痕跡検知 | `// Here's the solution:` 等の典型的な AI 出力痕跡を検出 | ○ |
| E | 自己説明テスト（Self-explanation） | 提出後にリーナが特定行の意図を質問し、答えられないと要注意 | ◎ |
| F | 段階提出チェックポイント | 長いコードを1発提出させず、数行単位で「ここまで見せて」を挟む | ○ |
| G | IDE 統合（VS Code 拡張等） | エディタ側で編集履歴・paste イベントを記録 | ❌ スコープ外 |
| H | タイピング速度 / クリップボード監視 | 技術的に不可能 | ❌ |

### 2.1 採用候補の詳細

#### E（自己説明テスト）— 最有力

- **仕組み**: 採点で合格判定を出す前に、リーナが 1〜2 箇所の行を指して「この行、自分の言葉で説明できますか？」と聞く
- **合否への影響**: 説明できなければ即不合格にはせず、**EXP を C/D ランクに抑える** + 「自分の理解で書けるよう、もう一度挑戦してみませんか？」と促す
- **長所**:
  - 学習効果そのものを高める（説明できる = 理解している、の原則）
  - コピペかどうかを直接判定するのではなく、「理解を確認する」形で自然に統合できる
  - 既存の採点フローに追加できる（新規インフラ不要）
- **短所**:
  - 採点ターンが1つ増える
  - 説明を AI が評価する必要があり、ゆるすぎる/厳しすぎるチューニングが難しい
- **実装コスト**: 小〜中（SKILL.md の採点フローに質問ステップを挿入、ボス判定では必須・通常演習では応答短縮など段階化）

#### C（カリキュラム先取り検知）— 補助的採用

- **仕組み**: カリキュラムの `lecture_topics` で扱った範囲のみを許容構文とし、未学習構文が出現したら採点時に指摘
- **長所**:
  - 論点が明確（「まだ習っていない構文が出ています」）で冒険者が納得しやすい
  - コピペの有無にかかわらず、学習順序を守る誘導になる
- **短所**:
  - 許容構文リストの整備コスト
  - Java は import や言語機能で境界が曖昧（`String.format` は標準的だが学習順では後回し等）
- **実装コスト**: 中（カリキュラムへ「許容構文」フィールド追加、採点 AI への指示）

#### D（AI 生成物の痕跡検知）— 軽量フィルタとして採用

- **仕組み**: 提出コードに ChatGPT/Claude の典型的な出力パターンが残っていないかを採点前に軽くスキャン
  - 例: ` // Here's the solution:`, `// This code ...`, `Certainly! Here's ...`, 過剰な docstring、説明的な英文コメントの連続
- **長所**: 一瞬の文字列検査で済む。誤検知時も「コメントを整理しましょう」で済み学習体験を損なわない
- **短所**: 高度な利用者は容易に回避できる。抑止力は弱い
- **実装コスト**: 小（正規表現リストで実装可能）

#### F（段階提出チェックポイント）— ボス戦のみ採用候補

- **仕組**: ボス問題だけは、一発提出でなく「最初のメソッド → チェック → 次のメソッド → ...」と段階的に提出させる
- **長所**: 冒険者がコードの構造を自分で考えている証拠になる
- **短所**: UX が重くなる。通常演習まで適用するとストレス
- **実装コスト**: 中（ボス戦フローの再設計）

### 2.2 不採用案の理由

| # | 案 | 不採用理由 |
|---|-----|-----------|
| A | 所要時間ヒューリスティック | 早書きの冒険者を誤検知する。トイレ離席等で長時間化する場合もあり、閾値設計が困難。学習体験を損なうリスクが効果に見合わない |
| B | 様式一貫性 | 学習が進めば当然 idiomatic になる。成長と検知の境界が曖昧で誤検知が多い。Eで理解確認できれば十分 |
| G | IDE 統合 | 別アプリ / 拡張の開発が必要。本issueのスコープを大きく超える。将来の拡張候補として保留 |
| H | キーストローク監視 | 技術的に不可能、かつプライバシー的に許容不可 |

## 3. 推奨アプローチ（CEO判断材料）

### 3.1 コア戦略: 検知ではなく「理解の確認」に振る

直接の「コピペ検知」は不可能かつ効果薄なので、**「自分で書いたコードは説明できる」という学習原理を組み込む** 方向を推奨する。コピペをした冒険者は自然に減速し、自力で書いた冒険者はスムーズに進める設計。

### 3.2 採用推奨の組み合わせ

| 層 | アプローチ | 適用範囲 |
|---|-----------|---------|
| 1. 軽量フィルタ | **D** AI 生成物痕跡検知 | 全提出 |
| 2. 学習順序ガード | **C** カリキュラム先取り検知 | 全提出 |
| 3. 理解の確認 | **E** 自己説明テスト | 通常演習（軽め）+ ボス（必須） |
| 4. 段階提出 | **F** チェックポイント | ボス戦のみ |

すべて「善意推定」— コピペ判定を宣言する処理は一切入れず、**学習行動として自然に見える形** で組み込む。

### 3.3 段階的導入プラン（案）

| フェーズ | 内容 |
|---------|------|
| Phase 1（軽） | D + E（ボスのみ）を先行導入 |
| Phase 2（中） | C をカリキュラムの「許容構文」整備と共に追加 |
| Phase 3（重） | E を通常演習にも拡大、F をボス戦に導入 |

Phase 1 だけでもコピペへの心理的抑止力としては相当機能すると予想。

## 4. 未解決課題（オープン）

1. **許容構文リストの設計コスト** — C の実装には各ダンジョンの「使ってよい構文」を明示する必要がある。どこまで細かく定義するか要議論
2. **自己説明テストの合否境界** — E で「説明が甘い」判定をどこに置くか。厳しすぎると学習体験を壊す
3. **通常演習への E 適用頻度** — 毎回聞くとうるさい、毎回聞かないと抑止力が弱い。ランダム or 怪しいときのみトリガーが良いか
4. **統計の蓄積と改善** — 効果測定のため自己説明応答の合否率を `sessions.log` に残すべきか（プライバシー考慮）

---

## 5. 実装着手時の注意事項

1. **冒険者を「コピペ疑惑」と明示的に扱わない** — リーナは常に「理解の確認をさせてください」という姿勢
2. **誤検知時の復帰導線を必ず用意** — 自己説明で追加情報を提供できれば EXP が戻る等
3. **他のフィードバック機能との重複チェック** — #1-3（コメント記法）/#1-4（命名規則）の採点項目と重ならないようバランスをとる
4. **採点AI のプロンプトにテンプレートを用意** — E / C / D の判定基準を明文化し、採点のブレを減らす
