策定日: 2026-04-11
策定者: セオ（devチーム テックリード）

## 0. 設計方針

改竄検知は **学習の誠実さを守る仕組み** であり、冒険者を罰する仕組みではない。
以下の3原則に基づいて設計する。

1. **善意推定**: 不整合の多くは手動編集ミスやファイル破損であり、悪意のある改竄は少数派と見なす
2. **最小リセット**: 不正が確認された範囲のみ巻き戻す。無関係な進捗には一切影響しない
3. **学習体験の保護**: 検知メッセージはリーナの口調で柔らかく伝える。冒険者の学習意欲を削がない

## 1. 検知対象

### 1.1 ボスコード存在チェック（既存設計の詳細化）

| # | 不整合パターン | 説明 |
|---|---------------|------|
| B1 | **boss/フォルダが存在しない** | `status: cleared` なのにboss/ディレクトリ自体がない |
| B2 | **boss/にJavaファイルがない** | boss/はあるがJavaソースファイル(.java)が0件 |
| B3 | **boss/のJavaファイルが空** | ファイルは存在するが中身が空（0バイト）またはコメントのみ |
| B4 | **boss/のコードが差し替えられた疑い** | 後述のハッシュチェックにより検知（v1.1以降） |

### 1.2 EXP/レベル整合性チェック

| # | 不整合パターン | 説明 |
|---|---------------|------|
| E1 | **レベルとEXPの不一致** | `stats.level` が `stats.exp` から導出されるレベルと一致しない（levelsテーブルで検証） |
| E2 | **exp_to_nextの不整合** | `stats.exp_to_next` がlevelsテーブルの定義値と一致しない |
| E3 | **EXP上限超過** | `stats.exp` がカリキュラム上の理論最大値（全ダンジョンの全演習+ボスをmax EXPで通過）を超えている |
| E4 | **ダンジョンEXPと演習実績の不整合** | ダンジョンの `dungeon_exp` が `exp_required` 未満なのに `boss_defeated: true` |
| E5 | **称号の不一致** | `stats.title` がlevelsテーブルの当該レベルの称号と一致しない |
| E6 | **負の値** | EXP、レベルに負の値が入っている |

### 1.3 ダンジョン進捗の論理整合性チェック

| # | 不整合パターン | 説明 |
|---|---------------|------|
| D1 | **前提未達のダンジョンがcleared** | `prerequisites` が未クリアなのに `status: cleared` になっている |
| D2 | **存在しないダンジョンID** | カリキュラムに定義されていないダンジョンIDが `dungeons` に含まれている |
| D3 | **boss_defeated と status の矛盾** | `boss_defeated: true` だが `status` が `cleared` ではない、またはその逆 |
| D4 | **boss_defeated_at の不整合** | `boss_defeated: false` なのに `boss_defeated_at` が設定されている |
| D5 | **cleared数とEXPの大幅乖離** | クリアダンジョン数が多いのにEXPが異常に低い（またはその逆） |

### 1.4 職業マスター状態の整合性チェック

| # | 不整合パターン | 説明 |
|---|---------------|------|
| J1 | **スキル未取得で職業マスター** | 基本職のマスターに必要な `required_skills` の一部が未取得（対応ダンジョン未クリア）なのにマスター扱い |
| J2 | **前提職業未マスターで上級職解放** | 上級職の `prerequisites` の基本職が未マスターなのに上級職が解放済み |
| J3 | **存在しない職業ID** | カリキュラムに定義されていない職業IDが `jobs` に含まれている |
| J4 | **スキル取得済みなのに職業が未反映** | 全スキルが揃っているのに職業がマスター扱いになっていない（悪意ではないが補正対象） |

### 1.5 YAML構造チェック

| # | 不整合パターン | 説明 |
|---|---------------|------|
| Y1 | **YAMLパースエラー** | 構文エラーでprogress.yamlが読み込めない |
| Y2 | **必須フィールド欠落** | `version`, `stats`, `dungeons`, `jobs` のいずれかが欠けている |
| Y3 | **型の不一致** | 数値フィールドに文字列が入っている等 |

## 2. 検知ロジック（起動時チェックフロー）

SKILL.md STEP 4「改竄検知」の内部処理を以下の順序で実行する。

### 2.1 実行順序

```
Phase 1: YAML構造検証
  ├── Y1: YAMLパース
  ├── Y2: 必須フィールド存在確認
  └── Y3: 型検証
  ※ Phase 1 で失敗 → 修復フロー（後述）へ分岐。以降のPhaseは実行しない

Phase 2: 数値・マスターデータ整合性検証（カリキュラムとの照合）
  ├── E6: 負の値チェック
  ├── E1: レベル ↔ EXP整合性
  ├── E2: exp_to_next整合性
  ├── E5: 称号整合性
  ├── E3: EXP上限チェック
  ├── D2: 存在しないダンジョンID
  └── J3: 存在しない職業ID

Phase 3: ダンジョン進捗の論理検証
  ├── D1: 前提ダンジョン未達チェック
  ├── D3: boss_defeated ↔ status整合性
  ├── D4: boss_defeated_at整合性
  ├── E4: ダンジョンEXP ↔ boss_defeated整合性
  └── D5: cleared数 ↔ EXP乖離チェック

Phase 4: ボスコード実体検証（最もコストが高い）
  ├── B1: boss/フォルダ存在確認
  ├── B2: Javaファイル存在確認
  ├── B3: ファイル内容の空チェック
  └── (該当するダンジョンのみ) AI再判定

Phase 5: 職業整合性検証
  ├── J1: 基本職マスター条件チェック
  ├── J2: 上級職前提チェック
  └── J4: スキル充足 → 職業未反映の補正
```

### 2.2 各Phaseの詳細判定条件

#### Phase 1: YAML構造検証

**Y1 — YAMLパース**
- `progress.yaml` をYAMLとして読み込む
- パースエラーが発生した場合 → **修復フロー** へ

**Y2 — 必須フィールド**
- トップレベルに `version`, `java_version`, `stats`, `dungeons`, `jobs` が存在すること
- `stats` 内に `level`, `exp`, `exp_to_next`, `title` が存在すること

**Y3 — 型検証**
- `stats.level`: 正の整数
- `stats.exp`: 0以上の整数
- `stats.exp_to_next`: 正の整数 または null（最大レベル時）
- `dungeons`: マップ型
- `jobs.basic`, `jobs.advanced`: マップ型

#### Phase 2: 数値・マスターデータ整合性

**E1 — レベル ↔ EXP整合性**
```
判定条件:
  levelsテーブルから stats.exp に対応する正しいレベルを算出
  stats.level が算出値と一致しない → 不整合
```

**E3 — EXP上限チェック**
```
判定条件:
  理論最大EXP = Σ(全ダンジョンの exp_boss.max) + Σ(全ダンジョンの exp_exercise.max × 推定最大演習回数)
  推定最大演習回数 = ceil(exp_required / exp_exercise.min) + 余裕係数(3)
  stats.exp > 理論最大EXP × 1.2（20%マージン） → 不整合
```
※ 推定は厳密でなくてよい。明らかに異常な値を弾くのが目的。

**D2 — 存在しないダンジョンID**
```
判定条件:
  dungeons のキーがカリキュラムの dungeon.id に存在しない → 不整合
```

#### Phase 3: ダンジョン進捗の論理検証

**D1 — 前提未達チェック**
```
判定条件:
  各ダンジョンDについて:
    D.status == "cleared" のとき
    D.prerequisites の全ダンジョンが status: cleared であること
    1つでも未クリア → 不整合
```

**E4 — ダンジョンEXP ↔ boss_defeated整合性**
```
判定条件:
  boss_defeated: true のとき
  dungeon_exp < exp_required → 不整合
  （ボス挑戦にはexp_required到達が必須なため）
```

**D5 — cleared数 ↔ EXP乖離チェック**
```
判定条件:
  cleared_count = status: cleared のダンジョン数
  min_expected_exp = Σ(clearedダンジョンの exp_required × 0.5 + exp_boss.min)
  stats.exp < min_expected_exp → 不整合（EXPが少なすぎる）
```
※ これは直接リセット対象にしない。警告フラグとして記録し、他の不整合と複合した場合のみ対処する。

#### Phase 4: ボスコード実体検証

**B1〜B3 — ファイル存在・内容チェック**
```
対象: dungeons で status: cleared の全ダンジョン

1. boss/ ディレクトリの存在確認
2. boss/ 配下の .java ファイルの存在確認
3. .java ファイルのサイズが0でないことを確認

いずれかに失敗 → AI再判定トリガー
```

**AI再判定のトリガー条件と判定フロー**
```
トリガー:
  - B1/B2/B3 のいずれかに該当（コードが不在または空）
  - B4: ハッシュ不一致（v1.1以降）

判定フロー:
  1. boss/ 配下に有効なJavaコードが1つでもあるか確認
  2. ない場合 → 即座に不合格（AI判定不要）
  3. ある場合 → カリキュラムの learning_goals 全項目に対してコードを評価
     - 全 learning_goals を満たしている → 合格（状態維持）
     - 一部でも満たしていない → 不合格（リセット対象）
     - コードが不完全でAIが判断できない → 不合格扱い（リセット対象）
```

#### Phase 5: 職業整合性検証

**J1 — 基本職マスター条件**
```
判定条件:
  基本職がmastered扱いのとき
  required_skills の各スキルに対応するダンジョンが全て status: cleared であること
  1つでも未クリア → 不整合
```

**J4 — スキル充足 → 職業未反映の補正**
```
判定条件:
  基本職の required_skills に対応するダンジョンが全て cleared なのに
  職業がmastered になっていない → 自動補正（マスター状態に更新）
```
※ J4は改竄ではなく進捗記録の欠損。冒険者に有利な方向に補正する。

## 3. 検知後の対応

### 3.1 重大度分類

検知された不整合を3段階に分類し、対応を分ける。

| 重大度 | 分類 | 対応 |
|--------|------|------|
| **CRITICAL** | YAMLパースエラー、必須フィールド欠落 | 修復フロー（バックアップ＋再構築） |
| **WARN** | 進捗データの論理不整合 | 該当箇所のリセット＋通知 |
| **INFO** | 軽微な不整合、自動補正可能 | 自動補正＋通知 |

### 3.2 不整合パターン別の対応表

| パターン | 重大度 | リセット範囲 | 対応内容 |
|---------|--------|-------------|---------|
| **Y1** YAMLパースエラー | CRITICAL | — | 修復フロー（3.3参照） |
| **Y2** 必須フィールド欠落 | CRITICAL | — | 修復フロー（3.3参照） |
| **Y3** 型の不一致 | WARN | 該当フィールド | 正しい型にキャスト試行。不可能ならデフォルト値に戻す |
| **E1** レベル ↔ EXP不一致 | WARN | level, title, exp_to_next | EXPを正とし、レベル・称号・exp_to_nextを再計算 |
| **E2** exp_to_next不整合 | INFO | exp_to_next | levelsテーブルから正しい値に自動補正 |
| **E3** EXP上限超過 | WARN | stats.exp | 理論最大EXPに切り戻し。レベル・称号も再計算 |
| **E4** ダンジョンEXP不足でboss撃破 | WARN | 該当ダンジョン | `boss_defeated: false`, `status: in_progress` に戻す |
| **E5** 称号不一致 | INFO | title | 正しい称号に自動補正 |
| **E6** 負の値 | WARN | 該当フィールド | 0にリセット。レベルは1に戻す |
| **D1** 前提未達でcleared | WARN | 該当ダンジョン | `status: in_progress`, `boss_defeated: false` に戻す |
| **D2** 存在しないダンジョンID | INFO | 該当エントリ | エントリを削除（無害だが不要） |
| **D3** boss_defeated ↔ status矛盾 | WARN | 該当ダンジョン | statusを正とする。clearedならboss_defeated: true、それ以外ならfalseに補正 |
| **D4** boss_defeated_at不整合 | INFO | boss_defeated_at | boss_defeated: falseなら boss_defeated_at を削除 |
| **D5** cleared数 ↔ EXP乖離 | INFO | — | 他にWARN以上の不整合がなければ無視。ログ記録のみ |
| **B1〜B3** ボスコード不在/空 | WARN | 該当ダンジョン | AI再判定 → 不合格なら `status: in_progress`, `boss_defeated: false` |
| **J1** スキル未取得で職業マスター | WARN | 該当職業 | 職業マスターを取り消し |
| **J2** 前提未マスターで上級職解放 | WARN | 該当上級職 | 上級職解放を取り消し |
| **J3** 存在しない職業ID | INFO | 該当エントリ | エントリを削除 |
| **J4** スキル充足→職業未反映 | INFO | 該当職業 | 職業をマスター状態に自動補正 |

### 3.3 CRITICAL: YAMLパースエラー時の修復フロー

```
1. 壊れた progress.yaml を progress.yaml.broken.{timestamp} にリネーム（バックアップ保全）
2. ファイルシステムから実績を再構築:
   a. 各ダンジョンフォルダを走査
   b. boss/ 配下にJavaファイルが存在するダンジョン → AI再判定
   c. 合格したダンジョンのみ status: cleared として再構築
   d. 通常演習フォルダ（ex-NNN_*/）の README.md から採点結果を読み取り、EXPを再計算
3. 再構築した progress.yaml を書き出す
4. 冒険者に修復結果を通知
```

### 3.4 通知メッセージ

全てリーナの口調で伝える。冒険者を責めるニュアンスは一切入れない。

#### CRITICAL（修復フロー実行時）

```
「あれ……？ {名前}さんの冒険の記録帳に不具合があるようです。
 でもご安心ください。ダンジョンに残された痕跡から記録を復元しますね。

 ……復元が完了しました。
 以下のダンジョンは記録を確認できました:
   [復元されたダンジョン一覧]

 もし復元結果に心当たりのない点があれば、教えてくださいね。」
```

#### WARN（進捗リセット時）

```
「{名前}さん、冒険の記録を確認していたのですが、
 いくつか記録に不整合が見つかりました。

 [不整合の内容を簡潔に]

 安全のため、該当部分の記録を調整させていただきました。
 ボスの再挑戦が必要なダンジョンがあります:
   [対象ダンジョン一覧]

 大丈夫、きっとすぐに取り戻せますよ！」
```

#### WARN（EXP/レベル補正時）

```
「{名前}さんの経験値の記録に少し誤差がありました。
 正しい値に修正しておきますね。

 EXP: {旧} → {新}
 レベル: Lv.{旧} → Lv.{新}

 冒険に影響はありませんので、ご安心ください。」
```

#### INFO（自動補正時）

```
「記録の小さな不整合を自動修正しました。
 冒険への影響はありません。」
```
※ INFOは詳細を逐一報告しない。複数あっても1行にまとめる。

#### J4（冒険者に有利な補正時）

```
「あ、{名前}さん！ 記録を確認していたら、
 【{職業名}】のマスター条件を満たしていたのに記録が漏れていました。
 今、反映しましたよ。おめでとうございます！」
```

## 4. 将来のAPI連携時の考慮

### 4.1 現在の位置づけ

ローカル改竄検知は **java-quest #2（AWS API構築）** までの暫定措置である。ただし、以下の理由からAPI移行後も完全には廃止しない。

### 4.2 API移行後のアーキテクチャ

```
[API移行前: 現在]

  冒険者のローカル環境
  ┌─────────────────────────────┐
  │  progress.yaml（正本）       │
  │  boss/ フォルダ（コード実体）│
  │  改竄検知（本設計書）        │
  └─────────────────────────────┘

[API移行後]

  冒険者のローカル環境              AWS API
  ┌───────────────────┐    sync    ┌──────────────────────┐
  │ progress.yaml     │◄─────────►│ DynamoDB/RDS（正本）  │
  │ （ローカルキャッシュ）│           │ S3（ボスコード保管）  │
  │ boss/（コード実体） │           │ サーバーサイド検証     │
  │ オフライン検知      │           └──────────────────────┘
  └───────────────────┘
```

### 4.3 progress.yaml の役割変化

| 観点 | 現在（ローカルのみ） | API移行後 |
|------|---------------------|----------|
| **データの正本** | progress.yaml | サーバー側DB |
| **progress.yamlの役割** | 唯一の進捗保存先 | ローカルキャッシュ（オフライン利用・高速読み込み用） |
| **改竄検知の主体** | ローカルのAI判定（本設計書） | サーバーサイドのバリデーション |
| **ローカル検知の役割** | 唯一の防衛線 | オフライン時のフォールバック＋起動時の簡易チェック |

### 4.4 API連携を見据えた設計上の配慮

#### 4.4.1 チェックロジックの分離

本設計のPhase 1〜5を独立した検証関数として設計する。API移行後は以下のように再利用する。

- **Phase 1（YAML構造検証）**: ローカルキャッシュの健全性チェックとして存続
- **Phase 2〜3（数値・論理検証）**: サーバーサイドに移植。ローカルはオフライン時のみ実行
- **Phase 4（ボスコード検証）**: サーバーサイドのAI判定に置き換え。ローカルはファイル存在チェックのみ
- **Phase 5（職業検証）**: サーバーサイドに移植

#### 4.4.2 同期コンフリクトへの対処方針

API連携時、ローカルキャッシュとサーバーの状態が乖離する場面が想定される。

```
方針: サーバーを常に正とする（Server Wins）

1. 起動時にサーバーから最新の進捗を取得
2. ローカルキャッシュと差分があればサーバー側で上書き
3. オフライン中の進捗はローカルに蓄積し、次回オンライン時にサーバーへ送信
4. コンフリクト時はサーバー側の値を採用し、ローカルのみの進捗は再検証キューに入れる
```

#### 4.4.3 ボスコードのサーバー保管

API移行後、ボス提出コードはS3にもアップロードする。

- ローカルの `boss/` フォルダは引き続きコードの実体として残す（開発ワークスペースの一部）
- S3にはバージョン管理付きで保管し、改竄検知の正本とする
- ローカルのboss/コードとS3のコードのハッシュ比較で差し替え検知が可能になる

#### 4.4.4 v1.1 拡張予定: コードハッシュ記録

API移行前でもローカルで実装可能な拡張として、ボス合格時のコードハッシュを `progress.yaml` に記録する方式を検討する。

```yaml
# v1.1 拡張案: progress.yaml への記録例
dungeons:
  C1-01:
    status: cleared
    boss_defeated: true
    boss_defeated_at: "2026-04-15T10:30:00+09:00"
    boss_code_hash: "sha256:a1b2c3d4..."  # ボス合格時のコードから算出
```

- ハッシュはboss/配下の全Javaファイルの内容を連結してSHA-256で算出
- 起動時にboss/のコードからハッシュを再計算し、記録値と比較
- 不一致の場合 → B4（コード差し替え疑い）としてAI再判定トリガー
- ハッシュ自体もprogress.yaml内にあるため改竄可能だが、コードとハッシュの両方を正しく偽装するコストは高く、抑止力として機能する

## 5. 実装上の注意事項

### 5.1 パフォーマンス

- Phase 1〜3, 5 はファイル読み込みとロジック判定のみ。起動時間への影響は無視できる
- Phase 4 のAI再判定はコストが高い。**B1〜B3に該当するダンジョンのみ** を対象とし、問題のないダンジョンはスキップする
- AI再判定が多数発生する場合（3ダンジョン以上）は、全体をリセットする方が効率的。個別判定は3ダンジョンまでとし、超過分は一括リセット

### 5.2 バックアップ

- CRITICAL対応で progress.yaml を再構築する前に、必ず旧ファイルをバックアップする
- バックアップファイル名: `progress.yaml.broken.{YYYYMMDD-HHmmss}`
- バックアップは3世代まで保持。古いものから削除

### 5.3 検知ログ

改竄検知の実行結果は progress.yaml に記録しない（progress.yaml はゲームデータに専念させる）。代わりにリーナの通知メッセージとして冒険者に直接伝える。

API移行後は検知ログをサーバーサイドに送信し、分析基盤で集計可能にする。

### 5.4 テスト方針

改竄検知ロジックは以下のケースでテストする（手動テスト手順をリリース前に整備する）。

| テストケース | 期待結果 |
|-------------|---------|
| 正常な progress.yaml | 不整合なし。通常起動 |
| YAMLが壊れている（構文エラー） | CRITICAL修復フロー発動 |
| レベルだけ99に書き換え | E1検知。EXPからレベル再計算 |
| EXPを99999に書き換え | E3検知。理論最大値にカット |
| 未クリアダンジョンをclearedに書き換え（boss/なし） | D1またはB1検知。in_progressに戻す |
| boss/のJavaファイルを削除 | B2検知。AI再判定（コードなし → 即不合格） |
| boss/のJavaファイルを別のコードに差し替え | AI再判定。learning_goals判定で合否決定 |
| 職業だけマスター済みに書き換え | J1検知。マスター取り消し |
| 全て正常だが職業反映漏れ | J4検知。マスター状態に自動補正 |
