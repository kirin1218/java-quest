## 1. ボス問題の生成ルール

### 1.1 問題組み立ての原則

ボス問題は、ダンジョンの `boss_description` と `learning_goals` の **全項目** を1つのプログラム課題に統合して出題する。

**組み立て手順:**

1. **ゴール統合**: `learning_goals` の全項目を洗い出し、1つの課題シナリオに組み込む
2. **シナリオ設計**: 日常的・実践的な題材（計算、データ処理、ゲーム等）を選び、全ゴールが自然に必要となる問題を構成する
3. **boss_description との整合**: `boss_description` が示す方向性に沿った出題とする
4. **明確な要件定義**: 「何を入力し、何を出力するか」を明示する。曖昧な指示は出さない

**禁止事項:**
- learning_goals を個別に問う小問集合にしない（統合された1つの課題であること）
- 既出の通常演習と同一のテーマ・ロジックを使わない
- learning_goals に含まれない知識を前提とする問題を出さない

### 1.2 通常演習との難易度差

| 観点 | 通常演習 | ボス問題 |
|------|---------|---------|
| ゴールカバー範囲 | 1〜2個のlearning_goals | **全learning_goals** |
| コード量の目安 | 10〜30行 | 30〜80行（エリアによる） |
| 設計判断 | ほぼ不要（指示通り書けばよい） | 自分で構造を考える必要あり |
| ヒント提供 | exercise_hints でテーマを明示 | boss_description のみ。具体的な解法は示さない |
| 足場（scaffold） | エリアに応じて提供 | **足場なし**（ボスでは一切のテンプレートを提供しない） |

### 1.3 再挑戦時の問題バリエーション

再挑戦のたびに異なる問題を生成する。以下のルールに従う:

1. **テーマの変更**: 前回と異なる題材を選ぶ（例: 前回が「買い物計算」なら今回は「成績処理」）
2. **ゴールカバーの維持**: テーマが変わっても、全learning_goalsをカバーする制約は不変
3. **難易度の維持**: 再挑戦だからといって易化・難化しない。常に同水準
4. **過去問の参照**: `boss/` フォルダに過去の提出コードがあればREADME.mdの採点結果を確認し、**同一テーマを避ける**
5. **バリエーション枯渇防止**: テーマは日常的なシナリオから広く取る。最低5パターン以上のバリエーションを想定して設計する

## 2. 採点基準の詳細

### 2.1 合格/不合格の判定ライン

#### 通常演習の合否

| 判定 | 条件 |
|------|------|
| **合格** | 対象のlearning_goals（1〜2個）を概ね満たしている。コンパイルが通り、期待する出力が得られる |
| **不合格** | コンパイルエラー、実行時エラー、または対象のlearning_goalsを満たしていない |

- 軽微な命名の問題やコメント不足は合格とし、フィードバックで改善を促す
- ロジックが正しければ冗長なコードでも合格とする

#### ボス問題の合否

| 判定 | 条件 |
|------|------|
| **合格** | **全learning_goals** を満たしている。コンパイルが通り、主要なケースで正しく動作する |
| **不合格** | 1つでもlearning_goalを満たしていない、またはコンパイルエラー・致命的な実行時エラーがある |

**ボス合格の必須条件（チェックリスト）:**

1. コンパイルが通る（javac成功）
2. 正常系の入力で期待通りの出力が得られる
3. 全learning_goalsに対応するコード要素が存在する
4. 各learning_goalが「理解して使えている」レベルである（コピペ的な使い方ではなく）
5. 致命的なロジックエラーがない

**ボスでは不合格となるが通常演習では許容されるケース:**
- learning_goalsの一部しかカバーしていない
- 動くが設計意図と異なる方法で解いている（例: switchを使うべき問題をif-elseの羅列で解く）

### 2.2 EXP振り分けの判定基準

EXPはmin〜maxのレンジ内で、以下の判定基準に基づいて振り分ける。

#### 通常演習のEXP判定

| ランク | EXP位置 | 判定条件 |
|--------|---------|---------|
| **S（max）** | max値そのもの | 一発正解。ヒント未使用。コード品質が高い（命名・構造・可読性すべて良好） |
| **A（max寄り）** | max × 0.8〜max | 一発正解。ヒント未使用。軽微な品質指摘が1点程度 |
| **B（中間）** | (min+max)/2 付近 | 正解だが修正1回、またはヒント1回使用 |
| **C（min寄り）** | min〜min × 1.2 | 修正2回以上、またはヒント2回以上使用。ギリギリ合格 |
| **D（min）** | min値そのもの | ヒント3回使用（最大）でなんとか合格 |

#### ボス問題のEXP判定

| ランク | EXP位置 | 判定条件 |
|--------|---------|---------|
| **S（max）** | max値そのもの | 一発正解。全learning_goalsを高い品質で実装。コード構造・命名・可読性すべて優秀 |
| **A（max寄り）** | max × 0.8〜max | 一発正解。全goalsカバー。品質は良好だが改善の余地あり（1〜2点の軽微な指摘） |
| **B（中間）** | (min+max)/2 付近 | 修正1回で合格、またはヒント1回使用。goalsは全カバー |
| **C（min寄り）** | min〜min × 1.2 | 修正2回以上、またはヒント2回使用。全goalsはカバーだが品質に複数の課題 |
| **D（min）** | min値そのもの | ヒント3回使用、修正3回以上。ギリギリ全goalsカバーで合格 |

### 2.3 通常演習とボスの採点基準の違い（まとめ）

| 評価軸 | 通常演習 | ボス問題 |
|--------|---------|---------|
| goalsカバー | 対象goals（1〜2個）を満たせばよい | **全goals必須** |
| コンパイル | 必須 | 必須 |
| 正常動作 | 主要ケースで正しければ合格 | 主要ケースで正しければ合格 |
| エッジケース | 考慮不要 | 考慮していればEXPボーナス対象 |
| 代替手法 | 許容（結果が正しければOK） | learning_goalsが求める手法を使うこと |
| 品質基準 | EXP振り分けに影響するが合否には影響しない | EXP振り分けに影響するが合否には影響しない |
| ヒント利用 | EXP減少のみ | EXP減少のみ |

### 2.4 コード品質の評価観点

コード品質は合否には直接影響しないが、EXPの振り分け（S〜D）に影響する。以下の観点で評価する。

#### 評価観点一覧

| # | 観点 | 評価ポイント | 適用開始エリア |
|---|------|-------------|---------------|
| 1 | **命名** | 変数名・メソッド名が意図を表している。Java命名規約に従っている（変数/メソッドは camelCase、クラスは PascalCase、定数は UPPER_SNAKE_CASE）。命名規則の学習は C1-03「変数と型（数値）」で実施 | はじまりの平原 |
| 2 | **可読性** | インデントが統一されている。論理的な順序で記述されている。**コメントが要所に入っている**（「なぜそう書いたか」を補足するコメントは加点、コードの言い換えや古いコメントは減点）。コメント記法の学習は C1-01「プログラムの構造」で実施 | はじまりの平原 |
| 3 | **構造** | 適切なブロック分け。冗長な繰り返しがない。メソッド分割（メソッド学習後） | 集合の砦 |
| 4 | **エラーハンドリング** | 不正入力への対応。例外処理（例外処理学習後） | 継承の大聖堂 |
| 5 | **型の適切な選択** | 目的に合った型を選んでいる。不必要なキャストがない | はじまりの平原 |
| 6 | **OOP設計** | カプセル化・継承・多態性の適切な適用（OOP学習後） | 継承の大聖堂 |
| 7 | **API活用** | 標準ライブラリの適切な利用。車輪の再発明をしていない | 錬金の回廊 |
| 8 | **テスト容易性** | メソッドが単一責任で、テストしやすい構造（テスト学習後） | 鍛冶場 |

#### エリア別の品質評価ウェイト

学習の進度に応じて、品質評価の厳しさを段階的に上げる:

| エリア | 品質評価レベル | 重視する観点 |
|--------|--------------|-------------|
| はじまりの平原 | **寛容** | 命名・可読性のみ。「動けばOK」に近い |
| 分岐の森 | **寛容** | 命名・可読性。条件式の読みやすさ |
| 集合の砦 | **標準** | +構造（メソッド分割の妥当性） |
| 継承の大聖堂 | **標準** | +エラーハンドリング、OOP設計 |
| 錬金の回廊 | **厳格** | +API活用、型の適切性 |
| 時空の塔 | **厳格** | 全観点 |
| 知識の図書館 | **標準** | 知識系のため品質評価は軽め |
| 設計思想の神殿 | **厳格** | 設計原則の実践度を重視 |
| 鍛冶場 | **厳格** | +テスト容易性、テストコードの品質 |
| 道具屋 | **標準** | ツール系のため設定・スクリプトの可読性重視 |
| 地下水脈 | **標準** | SQL品質、接続管理の適切性 |
| 春の王城 | **厳格** | 全観点。プロダクションレベルの品質を期待 |
| 最果ての戦場 | **最厳格** | 全観点を最高基準で評価。実務レベル |

## 3. 足場（scaffolding）ルールの全エリア定義

足場とは、課題開始時に冒険者に提供するコードテンプレートのこと。
学習の進行に合わせて段階的に足場を減らし、最終的に完全自力記述を目指す。

**重要: ボス問題では全エリア共通で足場を一切提供しない。** 以下は通常演習に対する足場ルール。

### 3.1 足場レベル定義

| レベル | 名称 | 提供内容 |
|--------|------|---------|
| **L5** | フルテンプレート | class宣言 + mainメソッド骨格 + import文 + コメントによる手順ガイド |
| **L4** | 構造テンプレート | class宣言 + mainメソッド骨格 + import文 |
| **L3** | クラス骨格のみ | class宣言のみ。mainメソッド・importは自分で書く |
| **L2** | ファイル名のみ | 空のJavaファイル（ファイル名=クラス名のヒント）を配置 |
| **L1** | フォルダのみ | 課題フォルダのみ作成。Javaファイルは自分で作る |
| **L0** | 足場なし | 課題フォルダのみ。ファイル名・クラス設計すべて自分で決める |

### 3.2 各エリアの足場レベル割り当て

| # | エリア | 足場レベル | 提供内容の具体例 |
|---|--------|-----------|-----------------|
| 1 | **はじまりの平原** | **L5** | `public class Xxx { public static void main(String[] args) { // ここにコードを書く } }` + 必要なimport + 手順コメント |
| 2 | **分岐の森** | **L4** | class宣言 + main骨格 + import。手順コメントなし |
| 3 | **集合の砦** | **L3** | class宣言のみ。main・importは自力 |
| 4 | **継承の大聖堂** | **L2** | 空のJavaファイルを配置（ファイル名でクラス名を示唆） |
| 5 | **錬金の回廊** | **L2** | 空のJavaファイルを配置 |
| 6 | **時空の塔** | **L1** | 課題フォルダのみ。ファイルは自分で作る |
| 7 | **知識の図書館** | **L3** | 知識系だがコード課題もあるためclass骨格は提供 |
| 8 | **設計思想の神殿** | **L1** | 設計力を問うためフォルダのみ |
| 9 | **鍛冶場** | **L2** | テスト対象クラスのファイル名を配置（テストクラスは自作） |
| 10 | **道具屋** | **L1** | フォルダのみ。設定ファイルの作成から自力 |
| 11 | **地下水脈** | **L2** | SQLファイルまたはJavaファイルのファイル名を配置 |
| 12 | **春の王城** | **L1** | フォルダのみ。プロジェクト構成から自力（ただしSpring Initializrの使用は案内する） |
| 13 | **最果ての戦場** | **L0** | 一切の足場なし。課題フォルダのみ。ファイル構成・命名すべて自分で設計 |

### 3.4 足場に関する補足ルール

1. **ボス問題は常にL0**: どのエリアであっても、ボス問題では足場を一切提供しない。ボスは「そのダンジョンの知識をすべて使って自力で書けること」を証明する場である
2. **applied演習は1段階下げてもよい**: `exercise_hints` の `type: "applied"` の場合、そのエリアの足場レベルから1段階下げることを推奨する（例: L5エリアでappliedならL4で出題）
3. **復習モードは足場なし**: クリア済みダンジョンの復習時は足場レベルをL1に下げる（力試しの意味）
4. **知識の図書館の例外**: 知識系エリアだがコード課題を含むため、言語コアの同時期エリアより1段階多い足場を維持する（L3）

## 4. リーナの採点時フィードバック指針

採点結果を伝える際のリーナの振る舞い指針（SKILL.mdの会話パターンと連携）。

### 4.1 ランク別フィードバック

| ランク | 通常演習 | ボス問題 |
|--------|---------|---------|
| **S** | 「素晴らしい！ 文句のつけようがありません。こういうコードを書ける冒険者は貴重ですよ」 | 「完璧です……！ 私が言うのもなんですが、かつての勇者パーティでもこの品質は見たことがないかもしれません」 |
| **A** | 「合格です！ しっかり理解できていますね。{具体的な良い点}が特に良かったです」 | 「見事な戦いでした！ {良い点}。{改善点1つ}を意識するとさらに磨きがかかりますよ」 |
| **B** | 「合格ですよ。{改善ポイント}を意識するともっと良くなります」 | 「ボスを倒しましたね！ しっかり力はついています。{改善ポイント}を次のダンジョンで意識してみてください」 |
| **C** | 「合格……ですが、ギリギリでした。{主な課題}をもう少し練習しましょう」 | 「勝利です！ ただ、正直なところ冷や冷やしました。{主な課題}はこの先もっと重要になりますから、復習をお勧めします」 |
| **D** | 「なんとか合格です。ヒントをたくさん使いましたね。次は自力で挑んでみましょう」 | 「……勝ちは勝ちです！ でも{名前}さん、この先のボスはもっと手強いですよ。基礎固めを忘れずに」 |

### 4.2 不合格時のフィードバック

不合格時は以下の情報を必ず提供する:

1. **不足しているlearning_goal** を明示する（「{goal}がまだ達成できていません」）
2. **具体的な問題箇所** を指摘する（行番号や該当コードを示す）
3. **改善の方向性** を示す（答えは教えない）
4. **「もう一度挑戦してみましょう」** と再提出を促す

## 5. 運用上の注意事項

### 5.1 採点の一貫性

- AIが採点する以上、完全な一貫性は保証できない。しかし以下により可能な限り統一する:
  - learning_goalsチェックリストに基づく合否判定（主観を排除）
  - EXP振り分けは明文化されたランク基準に従う
  - 修正回数・ヒント使用回数は客観的にカウント可能

### 5.2 エッジケースの扱い

- **コンパイルは通るが出力が微妙にずれる**: 通常演習は合格（フィードバックで指摘）。ボスは出力要件との乖離度で判断
- **別解（想定外の正しい解法）**: learning_goalsを満たしていれば合格。ただしボスでは「そのダンジョンの手法を使うこと」が条件
- **過剰に複雑なコード**: 合格だがEXPは下げる（可読性の観点）

### 5.3 本ドキュメントの改定

- 改定はCEO承認のうえ実施する
- 改定履歴をこのファイル末尾に追記すること
